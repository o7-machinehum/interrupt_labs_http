It starts as a boring flash drive.

Plug Phantomdrive in, and the host sees a regular USB mass storage device. The first section behaves like normal storage. The rest stays hidden until the unlock pattern is written.

Phantomdrive Unlock Flow

  1. 01

    Mounts normally

    Phantomdrive first appears as a normal USB drive.

  2. 02

    Waits for the trigger

    Create a file called unlock.txt containing password:addyourpasswordhere.

  3. 03

    Intercepts the write

    The firmware prevents that password file from persisting on storage.

  4. 04

    Reveals the rest

    The remaining disk is remounted, and data is encrypted and decrypted in place.

Project Details

  • Open hardware:Hardware, firmware, software, and mechanical files are open source.
  • Easy updates:The CH569W firmware can be updated by pressing the reset button while inserting it into your PC. No programmer required.
  • KDF:The key derivation function is open.
  • Measured speeds:We've recorded 20 MB/s reads and 9 MB/s writes.
  • Designed in KiCad:You do not need to pay thousands to edit the design files.
  • UART:UART is available through TP1, TP2, and TP3.
  • Nondescript enclosure:The enclosure has no special markings.
  • SD card storage:An SD card is used for storage, so you determine the capacity.

FAQ

Questions and Answers

Why crowdfunding? Why not Kickstarter?

I want to know how many units to order for the first batch. That simplifies distribution and helps me understand demand. I have used crowdfunding platforms before; they take a cut, which raises the price for you.

How far along is the project?

The engineering is finished. I just need to know the order volume before placing the first batch.

Isn't this just VeraCrypt?

No. A normal encrypted volume exposes random-looking encrypted data. Phantomdrive first presents as a normal USB drive, then reveals the protected section only after the unlock trigger.

Is entering a password into a text file insecure?

The firmware watches for the unlock write and does not allow that write to persist to storage.

Why use an SD card?

The current prototype uses an SD card for storage because eMMC pricing and availability were not attractive for this revision. A future eMMC board spin is still possible.

Is it fully open source?

The goal is full openness across hardware, firmware, and mechanical files.
